End-to-End Flow

How a Parcel Moves Through the System

A plain-English walkthrough of every stage — from the truck arriving at the dock to the parcel landing in the right outbound chute.

Phase 1 Physical Capture Truck → Belt → Camera
🚛 Entry
Truck Arrives at Hub
Delivery vehicle pulls into docking bay. Parcel unloaded and placed on intake belt.
⚙️ Transport
Intake Conveyor Belt
Speed-controlled belt transports parcel. Separation enforced between items.
📷 Capture
Multi-Angle Camera Capture
4 cameras fire simultaneously — top and 3 sides — as the parcel passes through. No stopping.
🛡️ Compliance
Privacy Filter
Faces and licence plates automatically masked before any data is stored. GDPR & CCPA compliant.
Phase 2 AI Analysis Visual Fingerprinting in <50ms
🧠 AI Engine
Image Ingestion & Preprocessing
Raw images normalised and queued into the AI pipeline in real time.
🔬 Features
AI Vision Analysis
Shape, texture, colour and damage patterns extracted. A 768-dimension visual fingerprint is created in <50ms.
📊 Score
Confidence Scoring
Similarity measured against known profiles. Anything below threshold is flagged for review.
Phase 3 Identity Decision Match, Create, or Fallback
🔍 Lookup
Identity Resolution
Visual fingerprint searched against the identity database in under 10ms.
Outcome
✓ Match Found
Existing Parcel Recognised

Confidence ≥ 87%. Record linked, history updated, 'Identified' event logged. Existing ID returned.

+ No Match
New Parcel Created

No vector match found. New parcel ID generated instantly. 'New Identity' event logged.

⚡ DB Offline
Fallback Mode Active

Database unreachable. System treats parcel as new temporarily. Reconciliation happens automatically on reconnect.

Phase 4 Routing & Dispatch Local Rules → Cloud Escalation
📝 Update
Update Parcel Record
Current location, timestamp, and status added to the parcel's history chain.
🗺️ Route
Routing Decision
Local rules applied first. If no rule found, cloud routing engine consulted for complex decisions.
☁️ Sync
Cloud Synchronisation
Record uploaded to cloud. All hubs globally updated. Offline sites queue data until reconnected.
Phase 5 Physical Dispatch Parcel Sorted in <200ms Total
📦 Command
Sorting Instruction Issued
Diverter gate commanded within 200ms of initial camera capture. No stopping, no delays.
⚙️ Physical
Diverter Gate Activates
Pneumatic or belt diverter switches the parcel onto the correct outbound chute — Zones A through F.
✅ Complete
Parcel Sorted & Dispatched
Parcel lands in correct outbound area. Operator dashboard updated. Zero manual handling.
🖥️ Monitor
Operator Dashboard
Real-time traffic light health status, throughput metrics, and alerts — visible to operators globally.

Software Architecture
F-01 · Camera Array
Multi-Angle Image Capture

Four HDR cameras fire simultaneously as the parcel passes. No stopping required — continuous flow capture.

Cameras: 4× (top + 3 sides)
Frame Rate: 30fps · HDR
Assist: Infrared low-light
Trigger: Dock sensor → belt
F-02 · Privacy Layer
Anonymisation & Compliance

Automated masking applied before any image enters storage or processing pipelines.

Masks: Faces, licence plates
Standards: GDPR, CCPA
Log: Compliance event per parcel
Mode: Configurable on/off
F-03 · AI Vision
Feature Extraction Engine

CNN model extracts a 768-dimensional visual fingerprint from each parcel image in under 50ms.

Architecture: CNN (custom)
Output: 768-dim embedding
Features: Shape, texture, colour
Latency: <50ms per parcel
F-04 · Identity DB
Vector Identity Resolution

Approximate Nearest Neighbour search matches embeddings against the parcel identity store in under 10ms.

Search: ANN · cosine similarity
Threshold: 0.85 confidence
Latency: <10ms lookup
Store: Local + cloud replicated
F-05 · Routing Engine
Smart Routing Decision

Local rule evaluation first; cloud escalation for complex multi-hub routing. Default fallback always available.

Priority: Local rules first
Escalation: Cloud engine
Timeout: Default local sort
Zones: A–F configurable
F-06 · Dispatch
Physical Sort Execution

Diverter gate commanded within 200ms of camera capture. Pneumatic or belt diversion to correct chute.

Command: <200ms end-to-end
Divert: Pneumatic / belt
Chutes: Up to 6 zones (A–F)
Accuracy: 99.4% sort rate
F-07 · Cloud Sync
Cloud Synchronisation

Event data batched and uploaded. Offline sites queue locally and reconcile automatically on reconnect.

Mode: Batch · retry on fail
Offline: Local queue → auto sync
Image Cache: 72h TTL local
Protocol: Configurable endpoint
F-08 · Archival
Image Archival & Storage

Images stored to local NVMe cache. Storage failure is non-blocking — processing continues with a logged warning.

Storage: Local NVMe cache
TTL: 72 hours
On failure: Skip + warn log
Design: Non-blocking
F-09 · Dashboard
Operational Health Monitor

Real-time traffic light status polled from all services. Green/Yellow/Red visibility for operators before each shift.

Status: Green / Yellow / Red
Poll: Service health check
Alerts: Low confidence, offline
Access: Operator dashboard
Resilience & Fallback Flows
⚡ Storage Offline
Image Archival Resilience

Local storage unreachable → archival step skipped → warning logged → processing continues uninterrupted. Data loss accepted for uptime.

⚡ Identity DB Offline
Identity Fallback Mode

Database unreachable → system defaults to treating all parcels as new → temporary profiles created → reconciliation on reconnect.

⚡ No Cloud Connection
Offline-First Operation

No cloud endpoint configured → offline mode activated → all data marked local-only → zero external traffic → full local operation.
